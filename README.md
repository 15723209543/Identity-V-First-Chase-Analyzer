# Identity V First Chase Analyzer

> 第五人格监管者首抓分析器

## 项目简介

本项目是一个基于 **Visual Studio C++ / EasyX** 的《第五人格》监管者首抓分析工具。程序会根据当前地图、求生者出生位置、求生者角色、监管者出生位置和监管者角色，综合计算监管者开局更可能首追哪一名求生者，并给出对应的首追概率、计算系数、分析原因和行动建议。

程序采用图形化点击输入方式。启动后会打开 EasyX 窗口，用户通过鼠标选择地图、4 名求生者的位置和角色、监管者位置和角色，程序随后在右侧窗口显示完整分析结果，并将分析文本自动保存到 `result/` 文件夹。

本项目主要用于 C++ 面向对象程序设计、EasyX 图形界面练习、游戏数据建模、地图几何分析和简单策略推荐展示。本项目不是《第五人格》官方工具，分析结果不代表官方结论，也不保证完全符合任何版本的真实对局环境。

## 与上一版本的主要不同

相比上一版本，本版本的核心变化不是简单修改界面，而是把地图和距离计算方式进行了明显升级。

| 对比项 | 上一版本 | 当前版本 `v2.11` |
|---|---|---|
| 地图显示 | 使用简化区域图或简化方位块 | 直接显示 `data/map` 中的真实地图图源 |
| 位置选择 | 主要按区域选择，位置粒度较粗 | 直接在原图白色可走区域点击，位置粒度更细 |
| 距离计算 | 主要使用区域中心点直线距离 | 使用墙体网格进行路径搜索，距离更接近实际绕墙跑图 |
| 地图资源影响 | 主要根据区域牵制分、开阔度、电机价值判断 | 在区域分基础上加入窗、板等附近资源修正 |
| 点击合法性判断 | 主要判断区域编号或区域块 | 同时判断地图裁剪范围、白色可走像素、外墙外白底、几何墙体网格 |
| 地图数据 | 主要依赖 `map_data.h` 的区域数据 | 新增 `map_geometry_data.h`，包含墙体网格、窗、板和地图真实比例 |
| 地图图源 | 无完整原图显示或原图参与较少 | 新增 `data/map/` 九张地图图源 |
| 工具脚本 | 主要生成 Excel 数据表 | 新增地图几何数据生成脚本 |
| 操作流程 | 选错后回退能力有限 | 增加返回地图、撤销上一名、撤销当前位置等操作 |
| 结果输出 | 输出首追概率和原因 | 输出概率、计算系数、文字原因、行动建议，并自动保存 |
| 项目结构 | 基础类和分析模块为主 | 增加地图几何、图像判定、寻路、资源点计算相关模块 |

简单来说，上一版本更像“基于区域的首抓概率估计器”，当前版本更像“基于地图图源和几何路径的首抓分析器”。

## 功能说明

本项目主要实现以下功能：

1. **地图选择**
   - 支持 9 张排位常见地图。
   - 程序左侧显示用户提供的地图原图。
   - 程序内部根据地图几何数据判断点击位置、墙体、窗、板和距离。

2. **求生者信息输入**
   - 支持依次选择 4 名求生者的位置。
   - 支持依次选择 4 名求生者角色。
   - 地图上会标记 `S1`、`S2`、`S3`、`S4`。
   - 选中角色后，会在地图标记附近显示求生者名字。

3. **监管者信息输入**
   - 支持选择监管者出生点或当前所在位置。
   - 支持选择监管者角色。
   - 地图上会标记 `H` 表示监管者位置。

4. **返回与撤销**
   - 第 1 名求生者位置页可以返回地图选择。
   - 后续求生者位置页可以撤销上一名求生者。
   - 角色选择页可以撤销当前位置选择。
   - 避免选错后必须重启程序。

5. **首追概率计算**
   - 根据求生者角色价值、监管者追击风格、地图几何压力、双方寻路距离和角色克制关系进行综合评分。
   - 使用 softmax 将 4 名求生者的原始分转换为概率百分比。
   - 输出监管者对每名求生者的首追意愿。

6. **结果展示**
   - 右侧区域显示完整分析文本。
   - 输出每名求生者的能力指标、出生地、首追概率、计算原因和行动建议。
   - 输出监管者能力分析和对每名求生者的追击判断。

7. **结果保存**
   - 每次运行结束后自动生成分析结果文本。
   - 结果保存到 `result/` 文件夹。
   - 文件名按当前时间生成，不会覆盖上一次结果。

## 使用方法

程序所有输入均通过鼠标点击完成，不需要在终端输入。

输入顺序如下：

1. 点击本局地图。
2. 在左侧地图原图上依次点击 4 名求生者所在位置。
3. 每选择一名求生者位置后，再点击该求生者角色编号。
4. 在左侧地图原图上点击监管者出生点或当前位置。
5. 点击监管者角色编号。
6. 查看右侧完整分析结果。
7. 在 `result/` 文件夹中查看自动保存的结果文本。

撤销规则如下：

- 第 1 名求生者位置页点击“返回选地图”，可以重新选择地图。
- 后续求生者位置页点击“撤销上一名”，会删除上一名已确认的求生者。
- 角色选择页点击“撤销位置”，会回到当前角色的位置选择。
- 只有有效地图范围内、外墙内部的白色空地可以点选；点到墙体、建筑、窗、板、门图标、文字、图例、外墙外白底或图片空白边时，程序会要求重新点击。

结果文件命名格式示例：

```text
result/2026-06-14_15-30-20.txt
```

每次输出都会新建一个按当前时间命名的文件，不会覆盖历史结果。

## 类与对象关系说明

本项目整体采用面向对象结构设计，核心思路是将“数据对象”“地图对象”“分析模型”“图形界面”“输出模块”和“程序控制器”分开。

### 1. 程序入口与控制器

```text
main()
└── first_chase_app
    ├── easyx_ui
    ├── data_repository
    ├── first_chase_analyzer
    └── report_writer
```

`main.cpp` 中的 `main()` 函数只负责设置控制台编码并创建 `first_chase_app` 对象运行程序。这样可以让主函数保持简洁，具体业务流程交给 `first_chase_app` 处理。

`first_chase_app` 是程序控制器，负责串联完整流程：

1. 初始化 EasyX 界面。
2. 读取地图选择。
3. 读取 4 名求生者的位置和角色。
4. 读取监管者位置和角色。
5. 调用首追分析器计算结果。
6. 调用结果生成器生成文本。
7. 调用界面类展示最终结果。

## 采用的算法说明

本项目不是简单地用固定规则判断首追目标，而是综合使用了多种算法和计算方法。

### 1. 加权评分算法

程序会对每名求生者分别计算多个维度的分数，然后按权重合成原始分。

主要评分项包括：

| 分值 | 作用 |
|---|---|
| `character_score` | 衡量求生者自身是否值得首追 |
| `hunter_score` | 衡量监管者风格是否适合追该求生者 |
| `map_score` | 衡量当前位置是否容易被追 |
| `distance_score` | 衡量监管者到该求生者的寻路距离 |
| `counter_score` | 衡量监管者和求生者之间的克制关系 |

总分公式如下：

```cpp
raw_score = 50
    + character_score * 1.00
    + hunter_score * 0.80
    + map_score * 0.55
    + distance_score * 0.70
    + counter_score * 1.35;
```

其中，克制关系权重最高，角色指标和监管者风格次之，地图和距离作为辅助修正。

---

### 2. 角色指标评分算法

角色指标分主要考虑：

- 破译能力
- 牵制能力
- 辅助能力
- 救援能力
- 首抓价值
- 弱牵制标签

大致逻辑是：破译越高、牵制越弱、首抓价值越高，越容易被监管者优先追击。

示意公式：

```text
角色指标分 =
    破译压力
  + 首抓价值压力
  + 弱牵制压力
  + 辅助位压力
  + 救援位压力
  + 特殊标签修正
```

---

### 3. 监管者风格匹配算法

监管者风格分用于判断监管者的能力类型是否适合追某个求生者。

主要考虑：

- 监管者追击能力和求生者牵制能力的差值
- 监管者控场能力对破译位的压制
- 监管者守椅能力对救援位的影响
- 监管者综合强度修正
- 追击弱牵制角色的额外收益
- 追击强牵制角色的额外成本

例如，追击能力强的监管者更愿意抓弱牵制角色；控场和信息能力强的监管者更重视高破译角色。

---

### 4. 角色克制标签算法

程序为监管者和求生者设置了标签，例如：

```text
anti_mobility
anti_item
anti_heal
anti_decoder
area_control
long_range
teleport
anti_loop
decoder
mobility
item_dependent
healer
stunner
kiter
weak_kiter
```

当监管者标签和求生者标签形成克制关系时，会增加追击分数。

例如：

- `anti_mobility` 克制 `mobility`
- `anti_item` 克制 `item_dependent`
- `anti_heal` 克制 `healer`
- `anti_decoder` 克制 `decoder`
- `anti_loop` 克制 `kiter`

此外，部分特殊监管者和求生者类型还设置了额外修正，例如摄影师、疯眼、隐士对破译位的压制，或部分角色机制导致的追击收益变化。

---

### 5. 地图区域压力算法

地图区域压力主要来自 `map_data.h` 中的区域评分。

每个区域都有：

- `kite_score`：牵制舒适度
- `openness`：开阔程度
- `decode_value`：电机 / 转点价值

基础逻辑是：

```text
区域牵制越弱，首追分越高；
区域越开阔，首追分越高；
区域电机价值越高，压制收益越高。
```

示意公式：

```text
region_pressure =
    (10 - kite_score) × 1.8
  + (openness - 5) × 1.0
  + (decode_value - 5) × 0.9
```

---

### 6. 附近资源修正算法

当前版本不只看区域，还会看点击点附近是否有窗和板。

程序从 `map_geometry_data.h` 中读取 `windows` 和 `pallets`，将窗和板看作地图线段，然后计算点击点到这些线段的距离。

主要逻辑是：

- 离窗越近，说明求生者附近有可用牵制资源，追击风险降低。
- 离板越近，说明求生者附近有可用板区资源，追击风险降低。
- 窗和板的保护分会从地图压力中扣除。

示意逻辑：

```text
position_pressure = region_pressure - nearby_resource_score × 1.7
```

资源分有上限，避免一个点附近资源过多导致评分失真。

---

### 7. 地图图像识别与几何数据生成算法

当前版本使用 `tools/generate_map_geometry.py` 从地图图源中提取几何信息。

主要步骤包括：

1. 对九张地图图片设置实际对局区域裁剪框。
2. 将裁剪后的地图划分为 90 × 90 网格。
3. 根据像素颜色判断墙体区域。
4. 根据蓝色像素识别窗。
5. 根据绿色像素识别板。
6. 使用连通块算法合并同一段窗或板。
7. 将窗、板和墙体网格转换为 0-1000 相对地图坐标。
8. 生成 `include/map_geometry_data.h`，供 C++ 程序使用。

其中，窗和板识别使用的是颜色阈值和连通块统计；墙体识别使用灰度、颜色差和网格内墙体像素占比判断。

---

### 8. 点击合法性判定算法

为了避免用户点到墙体、文字、图例、外墙外空白或不可走区域，本项目采用多层点击判定：

1. 判断鼠标是否在地图画布范围内。
2. 将屏幕坐标转换为 0-1000 地图相对坐标。
3. 判断是否在地图有效裁剪范围内。
4. 判断原图对应像素是否接近白色可走区域。
5. 判断是否属于外墙外侧的大块白底。
6. 判断该点对应的几何网格是否为墙体或障碍。
7. 只有通过全部判定，才允许作为求生者或监管者位置。

这样可以尽量避免点到不可走区域，提升分析输入的合理性。

---

### 9. 最短路径 / 寻路距离算法

当前版本的距离不再使用区域中心点直线距离，而是使用地图网格进行寻路。

主要步骤：

1. 将监管者位置和求生者位置转换为 90 × 90 网格下的起点和终点。
2. 如果起点或终点落在墙体网格上，就用 BFS 思想寻找最近的可走网格。
3. 使用带优先队列的 Dijkstra 最短路径算法计算两点之间的路径距离。
4. 支持 8 个方向移动，包括上下左右和斜向移动。
5. 斜向移动时会检查两侧相邻格，避免从墙角斜穿过去。
6. 每一步距离根据地图估算真实宽高换算为米制距离。
7. 如果寻路失败，则退回到欧氏距离并乘以修正系数。

距离越远，监管者开局首追该求生者的意愿越低。

距离修正逻辑：

```text
distance_score = - path_distance_m / 10
```

---

### 10. Softmax 概率归一化算法

4 名求生者都会得到一个原始分 `raw_score`。为了将原始分转换成首追概率，程序使用 softmax 算法。

公式为：

```text
probability_i = exp((score_i - best_score) / 18) / sum(exp((score_j - best_score) / 18))
```

其中减去 `best_score` 是为了避免指数计算过大；除以 18 是为了控制概率差距，不让小分差被过度放大。

最终输出为百分比，表示监管者更可能首追哪名求生者。

---

### 11. 文本自动换行与滚动显示算法

右侧结果区内容较多，程序会对中文文本进行自动换行处理。

实现思路：

1. 按 GBK 中文字符宽度拆分文本。
2. 使用 EasyX 的文字宽度判断当前行能否继续加入字符。
3. 超出宽度时自动换行。
4. 结果页支持滚动查看，避免长文本显示不完整。

## 设计特点

### 1. 主函数较简洁

`main.cpp` 只负责设置编码并启动 `first_chase_app`，避免所有逻辑堆在主函数中。

### 2. 输入输出和算法分离

`easyx_ui` 只负责界面，`first_chase_analyzer` 只负责算法，`report_writer` 只负责结果文本，便于后续修改和维护。

### 3. 静态数据集中管理

求生者、监管者、地图区域和地图几何数据分别放在独立头文件中，便于后续更新角色、地图和评分。

### 4. 支持图形化点击输入

用户不需要在控制台输入编号，直接通过 EasyX 界面点击位置和角色即可完成输入。

### 5. 支持地图几何路径计算

当前版本能够根据墙体网格计算路径距离，而不是只根据直线距离判断远近。

### 6. 输出结果可保存

每次分析结果都会保存为独立文本文件，便于复盘和对比。

## 当前不足与未完成内容

当前版本核心功能已经完成，但仍有一些没有完成或可以继续完善的地方：

1. **没有自动读取 Excel 数据**
   - `data/identityv_scores.xlsx` 目前主要作为整理表。
   - 程序实际使用的数据仍写在 `include/survivor_data.h`、`include/hunter_data.h`、`include/map_data.h` 和 `include/map_geometry_data.h` 中。
   - 后续可以增加从 Excel、CSV 或 JSON 自动读取数据的功能。

2. **权重暂未提供图形化修改入口**
   - 当前评分权重写在 `src/first_chase_analyzer.cpp` 中。
   - 后续可以增加“权重设置界面”，允许用户手动调整模型参数。

3. **地图几何识别仍有主观修正**
   - 当前地图几何数据来自图片颜色识别和手动裁剪框。
   - 对部分地图的特殊道路、白底和图标仍可能需要人工修正。

4. **暂未支持对局录像或截图自动识别**
   - 当前需要用户手动选择角色和位置。
   - 后续可以扩展为读取截图、录像或 OCR 识别，但当前版本未实现。

5. **结果输出格式仍可扩展**
   - 当前主要输出 `.txt` 文本文件。
   - 后续可以增加 Markdown、HTML、CSV 或图片报告导出。

6. **跨平台能力有限**
   - 当前依赖 Windows 和 EasyX。
   - Linux、macOS 暂不支持直接运行。

7. **数据版本需要持续维护**
   - 《第五人格》角色、地图和版本环境会更新。
   - 本项目评分数据需要根据实际版本继续调整。
